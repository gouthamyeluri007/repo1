Whats New in Intex Solutions Wrapper API (Vcmowrap) Version 3.5c

 

Library Version: 3.5.30.2

 

For a schedule of expected Wrapper version support dates, click here.

 

 

Key New Functionality

 

        Increased availability of loan identifiers in CLOs LoanX and FIGI now available more consistently across loans as a result of ID cross-mapping

 

        Ability to forecast a refi for tranches issued with an unfunded component in CLOs, revolver tranches may now be applied a refi forecast

 

        Student loan status transition forecasting apply prepays and defaults based on the collateral segmentation assumed for the purpose of status transition forecasting

 

        Reference deal-level attributes in query script access to extensive deal-level detail when building stratification criteria to be used for the purpose of applying asset bucket-level forecasting assumptions

 

        Cashflow modeling enhancements to improve cashflow accuracy, particularly in CLOs, Aircraft, and Auto Lease deals

 

        Utility for obtaining version info get info on the Wrapper version in use or on the latest version available for download

 

 

wcmo_deal()

 

        Increased availability of loan identifiers in CLO deals. LoanX and FIGI are available more consistently across loans as a result of ID cross mapping. An application looking to avoid having cross-mapped loan identifiers populated in CLOs may set a new CROSSMAP_CLOLOANIDS flag in the list of flags accepted by the IGNORE_DESCRIPTIVE_INFO keyword.

 

 

wcmo_deal_info()

 

        Deal-level information:

 

o   LATEST_DEAL_UPDATE_DATE

This new option / keyword reports the as-of date of the latest update, just like the CDU_ASOF_DATE keyword, but if the deal is already paid down then the keyword reports the CDU as-of date corresponding to the month on which the deal paid off.

 

o   DEAL_MODE

This existing option now reports a new keyword, DEAL_MODE_DISPLAY_DATE. It is the display date associated with the particular deal, deal mode, settlement date, and available update, and is the most appropriate date to report along with the deal mode. This mirrors the date which INTEXcalc displays on the far left side of the status bar at the bottom of the application. It is typically based on either the CDUs deal payment date or collateral factor date for seasoned deals, the deal closing date for a deal loaded in Pricing, or the deal paid down date for a deal already paid down.

 

 

        Group-level information:

 

o   GRP_COLLECTION_ACCT

Whenever a GRP_COLLECTION_ACCT_BAL_PRINCIPAL amount is available to be reported, the new GRP_COLLECTION_ACCT_PCT_PRINCIPAL keyword reports the same principal collection account as a percentage of the total outstanding collateral balance (collateral balance assumed to be inclusive of the principal collection account).

 

 

        Deal-amendment-level information:

 

o   DEAL_AMENDED_TYPE

This new option / keyword reports the type of amendment, as inferred from the deal type and from the deal details that were amended. Currently, this keyword may report one of the following: REFINANCED, RESET, REISSUED, or AMENDED.

 

o   DEAL_AMENDED_ACTION_TYPES

This version of the library supports the ability to amend the deal business center. This existing option may therefore report a new action through the DEAL_AMENDED_ACTION_TYPES keyword: AMENDED_DEAL_BUSINESS_CENTERS. Along with information that the deal business center has been amended, the DEAL_AMENDED_ACTION_PARAMS_VALUES and DEAL_AMENDED_ACTION_PARAMS_PREV_VALUES keywords list (~ delimited) the business centers making up the deal business center both after and before the amendment.

 

 

wcmo_collat()

 

        A loans "Initial ARM Reset Floor" is now reported, when applicable, from the LOAN_INFO[ARM_INITPERFLOOR] keyword.

 

 

wcmo_extinfo()

 

        Option to produce a new flavor of CLO market value based measures (e.g., LT 90 at MV):

 

The EXTINFO_WANTED options TR_CREDIT_SUPPORT_CURR_BASIS, GRP_CREDIT_SUPPORT_CURR_BASIS, and RATING_LEVEL_DATA can be used to produce different flavors of market value impacted measures for CLOs based on how the ASSETS_TO_VALUE_AT_MKT option is set. This version accepts a new option to allow for market value to only be considered below a certain market price, while assuming par above that price. This may be useful for analysis that assumes that high quality loans are held to maturity while lower quality loans are subject to sale/liquidation at their market price.

The newly supported ASSETS_TO_VALUE_AT_MKT_ADDL_COND_PRICE_LT input keyword may be assigned a market price between 0 and 100. The additional condition instructs the utility to only value asset at market price below a specified price. This condition will apply to any assets that would otherwise be valued at the market price based on the ASSETS_TO_VALUE_AT_MKT that was set.


INTEXcalc will be providing a set of market value based fields where assets above 90 are priced at par while market value is considered for assets priced below 90.

 

This can be implemented by setting the input options as follows:

ASSETS_TO_VALUE_AT_MKT=ALL
ASSETS_TO_VALUE_AT_MKT_ADDL_COND_PRICE_LT=90

 

 

        Utility for obtaining Wrapper version info

 

New EXTINFO_WANTED option PRODUCT_VERSION is a utility for obtaining information on the Wrapper version in use (or on an explicitly specified version) and on the latest Wrapper version available for download.

 

Input keywords:

 

o   PRODUCT_TYPE

The product for which version info is to be reported. Accepts the value WRAPPER (default value) or SUBS.

 

o   PRODUCT_VERSION

The Wrapper version for which info is requested. Specify a Wrapper version in the form emitted from wcmo_inits DLL_PRODUCT_VERSION keyword. If not specified, information on the Wrapper version in use will be returned.

 

o   CDU_PATH

A path to the CDU directory. This input is required if the utility is called prior to using wcmo_deal(). Please note that version info comes from the CDU directory and pointing to a stale CDU path will result in stale version information being reported.

 

Output keywords: 

o   PRODUCT_TYPE

The product name for which version info is being reported,
 

o   PRODUCT_VERSION

The version corresponding to the info being reported.
 

o   PRODUCT_VERSION_STATUS

The status of the version is either LATEST (latest version available for download), PREVIOUS (a prior version that is still supported), or NOT_SUPPORTED. The status could also be NA.

 

o   PRODUCT_VERSION_ERROR_MESSAGE

An error message, if there was an issue returning version info for the given version.
 

o   PRODUCT_VERSION_RELEASE_DATE

Release date of the version.
 

o   PRODUCT_VERSION_SUPPORT_END_DATE

Support end date of the version.

o   PRODUCT_VERSION_SUBS

The version of the core Subroutines embedded in the Wrapper version. This is similar in format to the ICMO_VERSION reported from wcmo_init().
 

o   PRODUCT_VERSION_LATEST

Latest version of the product available for download.
 

o   PRODUCT_VERSION_LATEST_RELEASE_DATE

Release date of the latest version available for download.

o   PRODUCT_VERSION_LATEST_SUPPORT_END_DATE

Support end date of the latest version available for download.
 

o   PRODUCT_VERSION_LATEST_SUBS

The version of the core Subroutines embedded in the latest Wrapper version available for download.

Please note that clients licensing the Wrapper that also have an Intex Web Services account may instead get VCMOWRAP library version information from the new IWS Product Version utility.

o   DEAL_COLLAT_TYPE

 

        Access to many more deal-level fields in a query script through the WDEAL_INFO() function:

 

The new WDEAL_INFO() function that may be referenced in query scripts provides access to any keyword available from deal-level wcmo_deal_info() calls (i.e, does not offer tranche-level, group-level, etc. wcmo_deal_info information).

 

Usage:

 

WDEAL_INFO(deal_info_wanted, [keyword], [data_type])

where

The deal_info_wanted argument could be any single option thats accepted through the DEAL_INFO_WANTED keyword in a deal-level wcmo_deal_info() call.

The keyword argument could be any wcmo_deal_info() output keyword corresponding to the specified deal_info_wanted option. By default, the requested keyword is assumed to be the same as deal_info_wanted.

data_type can be DECIMAL, INTEGER, STRING, NONE, DATE, and LIST. By default, STRING is assumed.

Example:

WDEAL_INFO(COLLATERAL_MANAGER,COLLATERAL_MANAGER_STANDARD)

returns the deal's standardized collateral manager.

Example:
WDEAL_INFO(COLLATERAL_MANAGER)

returns the collateral manager (not standardized), as the requested keyword is assumed to be the same as the deal_info_wanted.

 

Example:
WDEAL_INFO(DEAL_INTRINSIC_CURBAL)

returns a string version of the intrinsic balance (e.g. "5000000"). This means that an application will not be able to compare this value numerically.

Example:
WDEAL_INFO("DEAL_INTRINSIC_CURBAL","DEAL_INTRINSIC_CURBAL","DECIMAL") 

 

Returns the balance in numeric form (e.g., 5000000.00). It could then be compared numerically within the query script.

The data type of NONE may be used to test whether a value exists.

Example:
WDEAL_INFO("TRUSTEE_INDENTURE","TRUSTEE_INDENTURE","NONE")

returns 1.0 if no TRUSTEE_INDENTURE exists or 0.00 if it does.

 

Handling keywords representing lists:


Similar to WCOLL(), LIST data types are supported with the LIST_N, LIST_ITEM, and LIST_MATCH.

LIST_N may be used to get the number of elements in a list.

Example:
WDEAL_INFO("FINANCIAL_ADVISOR","FINANCIAL_ADVISOR","LIST","LIST_N")

Example:
Get the first element of the `FINANCIAL_ADVISOR' list with the following (the offset is an integer with no quotation marks):

WDEAL_INFO("FINANCIAL_ADVISOR","FINANCIAL_ADVISOR","LIST","LIST_ITEM",0)

 

Additionally, a LIST_ITEM could be specifically formatted with STRING, INTEGER, DECIMAL, DATE, or NONE.

Example:
WDEAL_INFO("MOST_SUBORD_JUN_SUB_TRANCHES","MOST_SUBORD_JUN_SUB_TRANCHE_LEVERAGE","LIST","LIST_ITEM",0,"DECIMAL")

Please note that if an application specifies an element beyond the LIST length (e.g. list has 2 elements, but item #4 is requested ) and the data type is not NONE, a blank string is returned if the data type is STRING (or not specified) or 0 is returned if numeric. If the data type is NONE, then a 1.0 is returned if data is not available and a 0.0 is returned if it is.

 

 

wcmo_cashflow()

 

        A refi forecast is now supported for CLO tranches issued with an unfunded component (revolver tranches). For such tranches, TR_DEAL_RESET_FCST_SUPPORTED reported from wcmo_deal_info() will return the value 1. Any refi assumptions applied to a revolver tranche (DEAL_RESET_TR_xxx keywords) will apply to the funded portion of the tranche. Furthermore, applying a tranche re-size (DEAL_RESET_TR_BALANCE keyword) will set the unfunded balance of the tranche to zero (tranche will be considered fully drawn as of refi).

 

        Applications may now apply assumptions such as prepayments and defaults based on the collateral segmentation assumed for the purpose of student loan status transition forecasting. Student loans are segmented when status transition forecasts are applied based on the number of components per initial student loan status called for by the forecast. Intex now tags each forecasted segment with a new _SPLT_STDNTLN_STATUS_CHANGE_FCST collateral field that can be referenced within a query script using the WCOLL() function.

 

Example:

STDNTLN_STATUS_CHANGE[SCHOOL][1].PERCENT=12.0

STDNTLN_STATUS_CHANGE[SCHOOL][1].FORECAST=CURRENT FOR 3 FORBEARANCE FOR 7

STDNTLN_STATUS_CHANGE[SCHOOL][2].PERCENT=4.0

STDNTLN_STATUS_CHANGE[SCHOOL][2].FORECAST=EMERGENCY_FORBEARANCE FOR 3

N_STDNTLN_STATUS_CHANGE[SCHOOL]=2

 

The above forecasts partitions all student loans in school into 3 segments:

A 12% segment forecasted to enter forbearance, a 4% segment forecasted to enter disaster forbearance. The remaining portion of the loan (84%) is not applied a transition forecast.

 

In this case, the Wrapper tags the 12% segment of each loan in school with _SPLT_STDNTLN_STATUS_CHANGE_FCST set to [SCHOOL][1] and the 4% segment of each loan in school with SPLT_STDNTLN_STATUS_CHANGE_FCST set to [SCHOOL][2]. A query script can apply a forecast to each segment by isolating them using WCOLL("_SPLT_STDNTLN_STATUS_CHANGE_FCST"). The remainder of the loans in school can be isolated in the query script by referencing WCOLL(STDNTLN_STATUS").

 

        A scenario multiplier may now be applied to the forbearance recovery scenario using the new FORBEARANCE_RECOVERY_SCENARIO_MULTIPLIER and FORBEARANCE_RECOVERY_SCENARIO_MULTIPLIER_TYPE keywords. When applying a scenario multiplier based on a query script, use a SCRIPT_PURPOSE of FORBEARANCE_RECOVERY_SCENARIO_MULTIPLIER_ASSET_QUERY for applying forbearance recovery multipliers.

 

The multiplier applies to forbearance recovery rates forecasted through FORBEARANCE_RECOVERY_RATE, FORBEARANCE_RECOVERY_OF_PREPAID_RATE, FORBEARANCE_RECOVERY_OF_DEFAULTED_RATE, and FORBEARANCE_RECOVERY_OF_MATURED_RATE. If FORBEARANCE_RECOVERY_ALSO_APPLIES_TO_DEFAULT is set as 0 then this multiplier does not apply to recovery from defaulted forborne balance.

 

 

wcmo_stats()

 

        In PRICEYIELDS and HORIZONRETURN modes, yield / spread / discount margin are computed using an iterative algorithm, and at times, when those measures are outside of an expected range, no solution can be found and the result is returned as NA. The API now offers additional detail in this situation to indicate whether the measure is greater than the upper tested limit or lower than the lower tested limit. When additional details can be offered to clarify NA stats results, the wcmo_stats() function emitted N_STATS_RESULT_DETAILS keyword will report a value thats greater than 0. That number represents the number of elements in the lists reported by the following new keywords:

 

o   STATS_RESULT_DETAIL_KEYWORD

The list of stats results for which additional detail is being provided. Currently, any of the following may be listed: YIELD, SPREAD_BP, NOMINAL_SPREAD, Z_SPREAD, DISC_MARGIN, and BE_EXPECTED_MARGIN.

 

In HORIZONRETURN mode, the following may also be listed: RETURN_YIELD, RETURN_DISC_MARGIN, RETURN_SPREAD, RETURN_Z_SPREAD, RETURN_HORIZON_YIELD, RETURN_HORIZON_DISC_MARGIN, RETURN_HORIZON_SPREAD, RETURN_HORIZON_Z_SPREAD, RETURN_HOLDING_YIELD, and RETURN_HOLDING_DISC_MARGIN.

 

o   STATS_RESULT_DETAIL_VALUE

Lists the limit value exceeded for each of the stats results that were returned as NA (as listed by STATS_RESULT_DETAIL_KEYWORD).

 

o   STATS_RESULT_DETAIL

For each limit value listed above, indicates whether the stat result would be expected above that limit (RESULT_ABOVE) or below it (RESULT_BELOW).

 

        The FACE_AMOUNT_ZERO_TREATMENT input keyword now accepts a new value, AS_ZERO. Setting this option indicates to the stats function that a zero or a small face amount (<= .01) should be considered as true 0 (i.e., zero cashflows). By default the API treats face amounts <=.01 as a flag indicating that full tranche face is to be used. Please note that the default face amount assumed in wcmo_stats() is 0, which typically indicates full face but with this option set to AS_ZERO would be interpreted as true 0.

 

        New by request asset-level cashflow vector LOAN_NDAYS_ACCRUED_INT reports the number of days in each projected accrual period.

 

 

General Improvements and Fixes

 

        Several cashflow modeling enhancements to improve cashflow accuracy, particularly in CLOs, Aircraft and Auto Leases deals.

 

        Improvement of known/historical cashflows for some Class pseudo tranches in cases where history is reported on the tranche-level but not on the class-level.

 

        A pricing curve node may now have maturity that is prior to settle date (or have a negative maturity, if specified using a number of months) when set via YIELD_CURVE_MATURITIES.

 

        Added support for floater index ceiling values (as reported by TRBLOCK_FLOAT_INDEX_CEILING) being zero or negative.

 

        The TRANCHE_INDEX_NAMES keyword reported from wcmo_deal() was adjusted to also include DERIVED indexes as well as INACTIVE indexes.

 

        ATTACHMENT_POINT, DETACHMENT_POINT, and THICKNESS cashflow vectors are now also reported for bonds with a 0 current balance that produce projected future cashflows.

 

        Adjusted how the WCOLL() query script function is evaluated on collateral that was applied a wcmo_collat_split(). WCOLL() now returns results for the original (pre-split) loan for all attributes other than for the specific tags intended for identifying the split components.

 

        Fixed the derivation of group-level DELINQ_CONSOLIDATED, DELINQ_30PLUS_FC_REO_INC_BK, DELINQ_60PLUS_FC_REO_INC_BK, and DELINQ_90PLUS_FC_REO_INC_BK for XRS groups.

 

        Now respecting the DECIMAL_DIGITS_SPEED setting when reporting the keywords associated with the following group-level in wcmo_deal_info() options: GRP_PREPHIST_PRICING, GRP_PREPHIST_CPR, GRP_1MO_CPR_HIST, and GRP_1MO_PRICING_HIST.

 

        Fixed that when producing a property report (wcmo_propinfo) based on a collateral group that include PNOTEs, some attributes were incorrectly reported.

 

        Now considering ratings (when available) by agencies beyond just S&P, Moodys and Fitch for the purpose of deriving data fields in EXTINFO_WANTED modes INTEX_SELECTED_TRANCHES and RATING_LEVEL_DATA and in wcmo_deal_info() reported TRIGGER_LOWEST_RELATED_RATING_ORIGINAL, DEAL_FIRST_RELATED_TRIGGERS, and DEAL_LAST_RELATED_TRIGGERS.

 

        Made a few adjustments to EXTINFO_WANTED mode COLLAT_AGGREGATES: now considering the XRS fraction when looking at collateral aggregates for an XRS group. No longer showing PAYSTATUS based aggregates for deals where theyre not applicable. Also adjusted the default ordering of subsets and fixed a crash.

 

        Any pipes (|) placed in a script model bucket name are now substituted with tildes (~) within LOAN_INFO[QUERYVAL] to avoid emitting a pipe-delimited list of an incompatible length.

 

        The following property balance fields are now properly scaled down to the PNOTE balance when COLLAT_LIMIT_TO_GROUP="TRUST": CURBAL, STRAT_CURBAL, PARTICPT_BALANCE_IN_TOP_TDC, and ORIGBAL.

 

        EXTINFO_WANTED mode RATING_LEVEL_DATA fields RATING_LEVEL_ATTACHMENT_POINT[<ratingLevel>], RATING_LEVEL_DETACHMENT_POINT[<ratingLevel>], and RATING_LEVEL_THICKNESS[<ratingLevel>] now report these measures using a cash basis.

 

        Now considering balloon and contract balloon extensions defined in template reinvestment assets (TBAs) to properly report the following wcmo_deal() keywords: DEAL_HAS_ANY_BALLOON, DEAL_HAS_BALLOONEXT_DEP, DEAL_HAS_CONTRACT_BALLOONEXT, DEAL_HAS_ANY_BALLOONEXT_FEES and DEAL_MAX_BALLOONEXT_CONTRACTS. Also avoiding a potential cashflow error when forecasting balloon extensions in a deal with contact balloon extensions in the reinvestment assets.

 

        wcmo_collat() reported LOAN_INFO[AMORT] ("Rate Type") now reports the referenced rate of a structured asset viewed in a top level collateral report. LOAN_INFO[AMORTTYPE_DETAILED] ("Amortization Type (for strats)") is no longer reported for structured assets.

 

        Fixed that a number of LOAN_INFOs were duplicated in the wcmo_collat() output when using the COLLAT_ORIENTATION=BY_COLLAT_ITEM option.

 

        Fixed a condition in which known index rates were not being applied to the forecast when back settling using USE_HIST_ALL=ALL in deals with missing updates (CDUs). Also addressed issues with using historical index rates in ARM loans.

 

        Fixed an issue in which an ARM coupon was being reset prior to its first forecasted payment date.

 

        Fixed an issue with a subset of tranches where the known index rate used for coupon calculation was a month off.

 

        Improved asset level accrual day calculation in a few deals.

 

        SCHEDPRINCIPAL and UNSCHEDPRINCIPAL cashflow vectors now available for tranches representing a "pass-thru" of the collateral. Specifically, this incudes tranches representing MBS pool and mega / giant pool cashflows.

 

        Fixed an issue in which EXTINFO_WANTED options TR_CREDIT_SUPPORT_CURR_BASIS and GRP_CREDIT_SUPPORT_CURR_BASIS results were unavailable for some specific CDOs.

 

        Fixed an issue where for multi-block tranches, block coupon caps and floors were reporting incorrect values in some cases.

 

        Fixed an issue with MODIFIED_AMORT and EXTEND_MATURED_NMONTHS forecasting where the forecast was being applied incorrectly in a deal loaded in WAVG mode and when setting DO_OPTIMIZE_CLUSTERING=0 at parse time. Also addressed a specific case where forecasting those assumptions caused an issue with subsequent cashflow runs.

 

        Fixed a case where the collateral gross margin was incorrect when exploding ARM mega / giant pools. Also now consistently removing 0 balance subpools from the collateral list when exploding ARM mega / giant pools.

 

        Fixed an issue in which setting single tranche mode was causing a difference in the tranche coupon depending on whether the deal was loaded directly or as a child deal backing a re-securitization.

 

        Fixed an issue applicable to some preprice deals where CDI_CREATE_DATE and CDI_CTRLNO were incorrect in situations where the creation date and control number of the CDI are unknown / unavailable.

 

        Forecasting a call with OPTREDEEM_ADDITIONAL_CONDITION_OVERRIDE set to SET_CALL_PRICE_TO_TARGET can no longer result in negative call proceeds being generated.

 

        Fixed some multi-threading issues, minor memory leaks, and a number of crashes. Also removed unnecessary parse & run-time errors.

 

        Implemented several speed ups.
